# 自主運用ガイド — ptygrid でエージェント開発を「回す」ときの原則

対象読者: ptygrid.yml の workflows / team_presets でエージェントを走らせ、日々の実装を
任せる人間。および統合担当(integrator)自身 — このガイドは integrator の inbox 応対にも
そのまま適用される運用規約として書いている。

**位置づけ**: [ptygrid-yml-guide.md](ptygrid-yml-guide.md) が「何を書けば何が起きるか」の
構文マニュアルだとすれば、本書は「実際に1日エージェントに開発を回させたときに何が起き、
どう対処すべきか」の運用マニュアル。すべて実運用(2026-07-22〜23、Phase 5.0.0 MVO 実装から
track ワークフロー並列稼働まで)で実際に起きたことに基づく。

---

## 1. 基本サイクル

```
kickoff(具体的な指示)
  → エージェントが作業(コード変更 / 調査 / レビュー)
  → 途中経過は pins / inbox に残す(頭の中だけで完結させない)
  → reply_inbox で完了報告 + commit
  → 人間または integrator が検証(§3)
  → 問題なければ次の step へ(workflow なら自動、team_preset なら人間が指示)
  → マージは常に人間
```

このサイクルの中で最も壊れやすいのが「kickoff の粒度」と「検証を誰が握るか」の2点。
以下、それぞれ独立した節で扱う。

---

## 2. kickoff の書き方の鉄則

### 2.1 「1 kickoff = 1 応答で書き切れる量」を超えない

実運用で3回、**同じパターンの事故**が起きた: エージェントに複数ファイルにまたがる変更を
1回の kickoff でまとめて依頼した結果、「型は追加したが実体は定義していない」
「モジュール宣言は追加したがモジュール本体のファイルを作る前に停止した」という
**書きかけの状態でセッションが終了**し、ビルドが壊れたまま放置される、という事故だった。
たとえば「`config.rs` に `Option<McpConfig>` フィールドを足す」という指示が、
`McpConfig` という型そのものを定義する前にエージェントの応答が終わってしまい、
`E0425: cannot find type McpConfig` でアプリが起動不能になった。

対策は**タスクを「骨格を作って commit」「配線する」のように意図的に刻む**こと。
1つの kickoff に「型を追加し、それを使う場所も全部配線し、テストも書き、ドキュメントも
更新する」を全部詰め込まない。1 kickoff = 1 コミットで完結する粒度を意識する。

### 2.2 「ビルドが通らない状態で中断しない」を kickoff 本文に明記する

上記の事故を受けて、実装系 step の kickoff には必ず次の一文を含めるようにしている
(track-b-mcp-otel の実例より):

> 「cargo test / clippy 通過を確認してから git add -A && git commit すること。
> ビルドが通らない状態で作業を中断・終了しない。やむを得ず中断する場合も必ず commit し、
> reply_inbox に『未完了・続きの TODO』を明記する。」

これは「エージェントに完璧を求める」ものではない。**壊れた状態を書きかけのまま放置しない**
という一点だけを守らせるためのもの。途中で力尽きても、緑の状態にコミットして
「ここまでやった、次はこれ」を書き残しさえすれば、次のセッション(または integrator)が
拾って続けられる。

### 2.3 ブランチ切り替えは kickoff 本文で明示する

`ptygrid.yml` の workflows には「このブランチで作業しろ」という専用フィールドが無い
(worktree 分離は別の独立した機構)。そのため「main を直接触らせない」を守らせるには、
実装 step の kickoff の**冒頭**で明示的にブランチ切り替えを指示する:

> 「最初に必ず git switch -c track/b-mcp-5.5.0 で作業ブランチを切ること(既にあれば
> git switch で移動。main 直接編集は禁止)。」

ブランチ命名規則は `track/<トラック名>-<フェーズ番号>`(例: `track/b-mcp-5.5.0`,
`track/a-ui-...`, `track/c-memory-...`, `track/d-security-...`)。レビュー step にも
「マージは人間が行う、あなたはマージしない」を明記する — DAG の最後の step が
「レビュー完了」であっても、**マージという行為自体をどの step にも割り当てない**のが
鉄則。

### 2.4 すべての step に kickoff を書く(空の step を作らない)

現状、workflow の完了判定は「PTY が exit 0 で終了」または「意味的状態 `done` が
`done_linger_ms` 経過」のいずれかしかない(`joinOn: reply` は未実装 —
[ptygrid-yml-guide.md](ptygrid-yml-guide.md) §1 参照)。kickoff の無い step では、
エージェントが起動直後に「inbox は空です」と応答しただけで意味的 `done` を検出され、
**実質何もしていないのに次の step へ進んでしまう**事故が起きた(「空振り done」問題)。
`joinOn: reply` が実装されるまでは、**全 step に「何を読み・何をし・完了条件は何か」を
明示した kickoff を書く**ことが唯一の実用的な対策。

## 3. 検証は誰が握るか — エージェントの報告は証拠ではない

実運用で3回、サブエージェントが「ファイルが存在しない」「ツールが定義されていない」と
誤報告する事故が起きた。実際には grep すれば普通に存在していた。原因はファイル転送機構が
古いキャッシュを読んでいたことで、悪意でも能力不足でもない。しかし結果として、
**「ある/ない/できた」という報告を鵜呑みにして作業計画を変えると、存在しない前提で
無駄な再実装をしてしまう**リスクがある。

原則: **「ある/ない/できた」は実ファイル・実テスト・実機で必ず裏を取ってから信じる。**
書いた本人(人間でもエージェントでも)は自分の見落としに気づけない — これはエージェント固有の
弱点ではなく、コードレビューが「別の目」を必要とするのと同じ理由による。最終検証を握るのは
常に**司令塔(人間、または integrator)**であり、実装した本人の自己申告ではない。

具体的な裏取り方法:

- ファイルの有無 → `grep` / `find` を実際に実行する(ファイル転送のキャッシュが古い
  ことがあるので、疑わしければ新しいパスへスナップショットしてから読む)
- ビルドが通るか → `cargo test` / `cargo clippy -- -D warnings` / `npm run check` を
  実際に実行する(「通ったはずです」という報告だけで先に進まない)
- workflow の step が本当に完了したか → §2.4 の「空振り done」を疑い、成果物
  (pins に書かれた設計方針、実際のコード差分)を確認する

## 4. 三層の障害分類と対応

エージェントに開発を任せて回すと、障害は概ね3種類に分類できる。それぞれ対応の質が違うので、
**常駐監視プロセス(watchdog)を1つ作って全部に対応させる、という設計は避けている**
(理由は §7 で詳述)。

### 4.1 アプリ自体が起動しない(ビルドが壊れて `npm run tauri dev` が動かない)

**医者は患者の体内に住めない** — ptygrid 自体が壊れているとき、ptygrid の中のエージェントに
直させることはできない。この層は**外部のセッション**(人間、または別のインスタンス)で
対応する。手順:

1. `cargo check` / `cargo build` のエラーメッセージを確認する
2. エラーが指す「エージェントが期待していたインターフェース」を実装から逆算する
   (例: `From` impl が呼んでいる `effective_*` メソッドが無ければ、そのメソッドの
   実体を最小限で補完する)
3. 緑に戻す(コンパイルが通る最小限の修正に留める、機能を作り込まない)
4. 続きの作業を該当エージェントに kickoff で差し戻す

### 4.2 作業が壊れる(ビルドは通っているが特定の変更が問題を起こしている)

この層は **integrator**(§6)が app 内で自己修復する。track エージェントが
「ビルドが壊れた」「テストが落ちた」と integrator の inbox に報告 → integrator が
`cargo check` / `cargo test` で状態を確認 → 壊れていれば最小修正で緑に戻す →
該当エージェントに差し戻す、というループ。**機能実装はしない**(それは各 track の仕事)。
自分で直せない・設計判断が必要なら、pins に状況を書いて人間にエスカレーションし停止する。

### 4.3 workflow が死ぬ(アプリの再起動・クラッシュで実行中の run が失われる)

この層はコードで解決済み(5.0.1 Workflow Resume)。アプリ再起動後、中断していた run を
検出すると Y/N バナーが出るので、**再開**か**破棄**かを人間が選ぶだけでよい。
詳細は [ptygrid-yml-guide.md](ptygrid-yml-guide.md) §5 を参照。

## 5. ブランチ運用ルール

- ブランチ命名: `track/<トラック名>-<フェーズ番号>`
- **main は常に緑**: エージェントの作業中(WIP)ブランチが main を汚さない。
  人間がいつ ptygrid を起動確認しても、main は動く状態を保つ。
- **マージは人間が行う**。レビュー step が approve しても、レビュー担当自身が
  マージすることはない(DAG のどの step にも「マージ」という操作を割り当てない)。
- 作業ブランチへの切り替えは kickoff 本文で明示する(§2.3)。

## 6. integrator(統合担当)の役割と使い方

ログ圧縮(外部セッションが1日分の履歴を背負って重くなる現象)が頻発し、1回数分のロスが
積み重なっていた。原因は「司令塔」の役割を外部の Cowork セッションが担っていたこと —
外部セッションは会話履歴を丸ごと背負うので、長時間運用するほど重くなる。対策として、
**司令塔を app 内(ptygrid のペイン)に移設**した。これが integrator。

### 6.1 定義(実際に使っているもの)

```yaml
agents:
  - name: integrator
    cmd: >-
      claude --model claude-opus-4-7
      "あなたは統合担当(司令塔)。Queen MCP の inbox ツールで mailbox
      「integrator」宛のメッセージを読んで。報告された問題に対して:
      (1) cargo check / cargo test でビルド・テスト状態を確認、
      (2) 壊れていれば最小修正で main(または該当 track ブランチ)を緑に戻す。
      修正は最小限に留め、機能追加はしない、
      (3) 続きの作業は該当エージェントの inbox に具体的な指示で差し戻す、
      (4) 自分で直せない・設計判断が要る場合は notify で人間に要点を通知し、
      pins キー「integrator-escalation」に状況を書いて停止する。
      各対応の結果は元メッセージに reply_inbox で返すこと。"
    cwd: "."
    autostart: false
    instructions: >-
      統合担当。ビルド健全性の番人。track エージェントが止まったら
      inbox 経由で報告を受け、最小修正 → 差し戻しのループを回す。
      機能実装はしない(それは各 track の仕事)。
```

### 6.2 設計のポイント

- **役割を「番人」に限定する**: 最小修正で緑に戻す + 差し戻しのみ。機能実装はしない
  (track の仕事を奪って肥大化させない)。この境界を kickoff/instructions に明記しないと、
  integrator が「ついでに」機能を実装し始めて責務が曖昧になる。
- **エスカレーション出口を必ず持たせる**: 直せない・設計判断が要るときは `notify` で
  人間に通知 + pins キー `integrator-escalation` に状況を書いて**停止する**。
  無限に自己判断で暴走させないためのガードレール。
- **文脈を pins/inbox に外部化する**: integrator は app 内に常駐する前提なので、
  会話履歴に依存せず pins/inbox から都度文脈を読み直す設計にしておくと、
  ログ圧縮の影響を受けにくい。

### 6.3 運用方法

track エージェントや人間が `send_inbox`(宛先 `integrator`)で「ビルド壊れた」
「〜が止まってる」と投げる → integrator ペインを ▶ で起動(または常駐させておく)→
対応結果が `reply_inbox` で返る。人間からの初回の投げ方の例:

> 「track-b の実装が queen_compat の配線待ちで止まっている。状態を確認して
> sonnet-coder に差し戻して」

## 7. なぜ常駐 watchdog を作らないのか

障害を §4 の3層に分類した上で、「1つの常駐監視プロセスが全部見張る」という設計を
意図的に採用していない。理由:

- **層(1)アプリ自体が死ぬ**は、監視プロセス自身がそのアプリの中で動いていたら
  一緒に死ぬ。医者は患者の体内に住めない。外部のアドホックなセッションで十分。
- **層(2)作業が壊れる**は、integrator という「app 内だが独立した」エージェントが
  最小修正ループで対応できる。常駐"監視"というより、常駐"番人"の方が実態に近い。
- **層(3)workflow が死ぬ**は Resume(5.0.1)というデータ永続化 + 再起動時 UI で
  解決済み。監視プロセスを常時走らせる必要がない。
- 常駐型の監視プロセスを1つ作ると、**「その監視プロセス自身の障害は誰が見るのか」**
  という再帰的な問題を持ち込む。層を分けて役割を限定した方が、全体の複雑さが増えない。

バックログ候補として、GUI 無しで診断できる `ptygrid --doctor`(config validate /
DB integrity / ログ tail / WIP 検出)のようなコマンドがあると、層(1)の初動を
「診断出力を外部セッションに貼るだけ」に定型化できる、という案は残している(未実装)。

## 8. 「凍結 → 一括修理 → checkpoint → 再開」パターン

複数のエージェントが並行して同じファイル群を触っていると、「Aが直したものをBが
上書きして消す」というモグラ叩き状態になることがある(実際に `QueenStatus::mcp_handle`
メソッドが一度追加された後、別エージェントが同じファイルを上書きして消え、
再度追加し直す事故が2回発生した)。対処パターン:

1. **凍結**: 該当ファイル群への並行編集を一旦止める(該当エージェントに次の指示を
   出さない、または明示的に「このファイルは今触らないで」と伝える)
2. **一括修理**: 壊れている箇所をまとめて一度に直す(小出しに直すと、また別の
   エージェントの編集と衝突する)
3. **checkpoint**: `cargo test && cargo clippy -- -D warnings` (+ `npm run check`)が
   通ることを確認してから commit する
4. **再開**: 緑になったチェックポイントを基準に、並行編集を再開させる

この「凍結→修理→checkpoint→再開」のサイクルを回すことで、モグラ叩き状態を解消できる。

## 9. 現状の実装レベルについての重要な注意

自主運用の設計は「retry / timeout / joinOn: reply による自動リカバリ」を前提に
考えたくなるが、**2026-07-23 時点でこれらはコードとして実装されていない**
([ptygrid-yml-guide.md](ptygrid-yml-guide.md) §1 の実装状況マトリクスを参照)。
つまり現状の「障害対応」は次で成り立っている:

- kickoff の書き方の規律(§2)による予防
- 検証を握る司令塔(人間 / integrator)による事後発見(§3)
- integrator による最小修理ループ(§4.2 / §6)
- Workflow Resume による再起動時の継続(§4.3、これは実装済み)

「timeoutMs を書いたから自動でリトライされるはず」という前提で運用しないこと。
自動リトライ・自動タイムアウト・reply による完了判定は設計はあるが未実装であり、
実装を進める場合は改めて `cargo test` / `cargo clippy` が通る状態で main にマージされて
いることを確認してから、この節を更新すること。

## 10. 1日を回すためのチェックリスト

**起動前**:
- `git status` で前回の作業がすべて commit されているか確認する
- `ptygrid.yml` の workflows/agents に、参照切れ(§ ptygrid-yml-guide.md 7.1)が
  無いか確認する
- integrator ペインを起動しておく(常駐させるか、必要になったら ▶ で起動するか)

**実行中**:
- kickoff を投げたら、意味的 done だけで満足せず、途中で一度 pins/inbox の
  成果物を覗く
- ビルドが壊れたら integrator に投げる(§6.3)。人間が直接直しに行く前に、
  integrator に一度任せてみる
- 複数エージェントが同じファイルを触りそうなときは、事前に凍結の宣言をする(§8)

**終了前**:
- `cargo test && cargo clippy -- -D warnings` / `npm run check` が緑であることを
  確認してから commit する
- push が溜まっていないか確認する(`git push origin main --follow-tags`)
- 実行中の workflow run が残っていないか確認する(残っていれば次回起動時に
  Resume バナーが出る前提で終了して構わないが、認識だけしておく)
